## How Can I Contribute?

### Reporting Issues

If you find a bug, error, or have a suggestion:

1. **Search existing issues** to avoid duplicates
2. **Use the issue template** provided
3. **Provide detailed information** including:
   - Clear description of the issue
   - Steps to reproduce (for bugs)
   - Expected vs actual behavior
   - Environment details (OS, software versions)
   - Screenshots if applicable

### Suggesting Enhancements

We welcome suggestions for:

- New lab exercises
- Additional datasets
- Improved documentation
- New automation scripts
- Framework updates
- Tool recommendations

Please use the feature request template and provide:
- Clear description of the enhancement
- Use case and benefits
- Potential implementation approach
- Any relevant examples or references

### Contributing Content

You can contribute:

1. **Lab Improvements**
   - Corrections to existing labs
   - Additional exercises
   - Updated instructions
   - Better explanations

2. **Documentation**
   - Typo fixes
   - Clarity improvements
   - Additional examples
   - Translation (if applicable)

3. **Resources**
   - New datasets
   - Additional scripts
   - Framework documentation
   - Tool guides

4. **Templates**
   - New templates
   - Improved existing templates
   - Additional examples

## Contribution Guidelines

### Before You Start

1. **Fork the repository** to your GitHub account
2. **Clone your fork** locally
3. **Create a branch** for your contribution
4. **Review existing content** to understand structure and style

### Making Changes

1. **Follow the existing structure**
   - Maintain directory organization
   - Use consistent naming conventions
   - Follow established formats

2. **Maintain professional standards**
   - Use clear, professional language
   - Avoid emojis or casual expressions
   - Ensure technical accuracy
   - Provide proper citations

3. **Test your changes**
   - Verify all links work
   - Test any scripts or code
   - Check formatting renders correctly
   - Validate datasets load properly

4. **Document your changes**
   - Update relevant README files
   - Add comments to code
   - Explain complex concepts
   - Provide usage examples

### Submitting Changes

1. **Commit your changes**
   ```bash
   git add .
   git commit -m "Brief description of changes"
   ```

2. **Push to your fork**
   ```bash
   git push origin your-branch-name
   ```

3. **Create a Pull Request**
   - Use the PR template
   - Provide clear description
   - Reference related issues
   - Explain the benefit

4. **Respond to feedback**
   - Address review comments
   - Make requested changes
   - Be open to suggestions

## Development Process

### Branch Naming

Use descriptive branch names:
- `fix/lab-week-5-typo`
- `feature/add-siem-dataset`
- `docs/update-readme`
- `enhancement/improve-risk-template`

### Commit Messages

Write clear commit messages:
- Use present tense ("Add feature" not "Added feature")
- Be specific and descriptive
- Reference issues when applicable
- Keep first line under 50 characters

Examples:
```
Fix typo in Week 10 lab instructions
Add compliance audit dataset for ISO 27001
Update NIST CSF framework documentation
Improve risk assessment template with examples
```

### Pull Request Process

1. **Ensure your PR:**
   - Addresses a specific issue or need
   - Follows contribution guidelines
   - Maintains code quality standards
   - Includes appropriate documentation
   - Passes all checks (if applicable)

2. **PR will be reviewed for:**
   - Technical accuracy
   - Alignment with programme objectives
   - Quality and clarity
   - Consistency with existing content
   - Educational value

3. **Review timeline:**
   - Initial review within 5 business days
   - Feedback provided if changes needed
   - Approval and merge when ready

## Quality Standards

### Content Quality

All contributions must meet these standards:

1. **Accuracy**
   - Technically correct information
   - Current industry practices
   - Proper framework alignment
   - Validated against standards

2. **Clarity**
   - Clear, concise language
   - Logical organization
   - Appropriate detail level
   - Good examples

3. **Professionalism**
   - Professional tone
   - Proper grammar and spelling
   - No casual language or emojis
   - Appropriate formatting

4. **Completeness**
   - All necessary information included
   - No broken links or references
   - Working code and scripts
   - Proper documentation

### Lab Exercise Standards

Lab contributions should include:

- Clear learning objectives
- Detailed instructions
- Required resources list
- Expected outcomes
- Assessment criteria
- Estimated completion time
- Difficulty level
- Prerequisites

### Dataset Standards

Dataset contributions should:

- Use realistic but fictional data
- Include data dictionary
- Provide usage examples
- Specify format and structure
- Include source attribution (if applicable)
- Be properly formatted
- Include validation checks

### Documentation Standards

Documentation should:

- Follow Markdown best practices
- Use consistent formatting
- Include table of contents (for long docs)
- Provide examples where appropriate
- Link to related resources
- Be well-organized
- Use proper headings
